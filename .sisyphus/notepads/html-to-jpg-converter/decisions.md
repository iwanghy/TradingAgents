# HTML to JPG Converter - 架构决策

## 2026-03-17 - 章节分割功能架构

### 决策 1: 分段策略
**选择**: 按逻辑章节分割,而非固定高度/字符数

**原因**:
- 保持语义完整性,每个 section 是一个独立的分析单元
- 避免在段落中间分割
- 符合 HTML 结构设计意图

**权衡**:
- 优点: 语义清晰,易于理解
- 缺点: 每个 section 长度不一,可能导致某些页面过短或过长

### 决策 2: 临时文件处理
**选择**: 使用 NamedTemporaryFile + 主动清理

**原因**:
- 确保进程异常退出时也能清理临时文件
- 避免临时文件累积
- 跨平台兼容性好

**替代方案**:
- 使用 /tmp 目录 + 手动命名: 容易产生文件名冲突
- 使用内存文件系统: 复杂度高,跨平台兼容性差

### 决策 3: HTML 片段构建
**选择**: 为每个片段构建完整的 HTML 文档

**原因**:
- imgkit.from_file() 需要完整的 HTML 文档
- 保持样式一致性 (每个片段包含相同的 style 标签)
- 避免样式丢失或渲染异常

**权衡**:
- 优点: 渲染稳定,样式一致
- 缺点: 增加了 HTML 体积 (但影响可忽略)

### 决策 4: 错误处理策略
**选择**: 验证 HTML 结构 + 清晰的错误消息

**原因**:
- 提前发现结构问题,避免运行时错误
- 帮助用户快速定位问题
- 符合 Python 最佳实践

**实现**:
```python
if not header:
    raise ValueError("HTML 缺少必要的结构元素: header")
```

### 决策 5: 向后兼容性
**选择**: 默认 `enable_segmentation=False`,保持现有行为

**原因**:
- 不破坏现有代码
- 用户可选启用新功能
- 符合"最小惊讶原则"

### 决策 6: 最大片段数量限制
**选择**: 支持 `max_segments` 参数,默认值为 5

**原因**:
- 防止生成过多图片导致性能问题
- 给用户控制权
- 大部分报告不超过 5 个片段

**实现**:
```python
if len(html_segments) > max_segments:
    html_segments = html_segments[:max_segments]
```
